# 프롬프트 작성 가이드

이 가이드는 Harness 프레임워크의 스킬 작성 원칙을 기반으로, 각 챕터 전문가가 자기 스테이지의 SKILL.md를 고도화할 때 참조한다.

## 디렉터리 구조

```
prompts/
├── base.system.md              # 전역 지시사항 (전 스테이지 공통)
├── policy.md                   # 비즈니스 정책 SSOT
├── GUIDE.md                    # 이 문서
├── agents/                     # 에이전트 페르소나 (누가)
│   ├── pm.md
│   ├── ux.md
│   ├── pd.md
│   ├── fe-dev.md
│   ├── be-dev.md
│   └── qa-eng.md
├── context/                    # 아키텍처 컨텍스트 (조건부 로딩)
│   ├── fe-architecture.md
│   └── be-architecture.md
└── skills/                     # 스테이지별 스킬 (어떻게)
    └── {stage}/
        ├── SKILL.md            # 핵심 절차 + 규칙
        ├── template.md         # 출력 스키마/예시
        ├── scripts/verify.sh   # 출력 검증 스크립트
        ├── references/         # 조건부 참조 문서 (선택)
        └── assets/             # 정적 자산 (선택)
```

## 핵심 원칙: 에이전트와 스킬의 분리

| 구분 | 에이전트 (`agents/`) | 스킬 (`skills/`) |
|------|---------------------|-----------------|
| 정의 | **누가** — 전문가 페르소나 + 판단 원칙 | **어떻게** — 절차적 지식 + 출력 규칙 |
| 내용 | 역할, 작업 원칙, 판단 기준, 입출력 프로토콜 | 스테이지별 절차, 출력 포맷, 검증 규칙 |
| 변경 주기 | 드묾 (역할 정의 변경 시) | 잦음 (품질 개선, learn 반영) |

에이전트 파일에는 "무엇을 만들어라"를 넣지 않는다. 스킬 파일에는 "어떤 관점으로 판단하라"를 넣지 않는다.

## SKILL.md 작성 규칙

### 1. YAML Frontmatter 필수

```yaml
---
name: stage-name
description: "적극적(pushy) 설명 — 이 스킬이 무엇을 하고, 어떤 입력에 트리거되며, 어떤 형태로 출력하는지를 한 문단으로."
---
```

description은 스킬의 트리거 메커니즘이다. 짧고 모호한 설명은 피하고, 구체적인 입력 유형/출력 형태/적용 범위를 명시한다.

나쁜 예: `"요구사항을 정리하는 스킬"`
좋은 예: `"비정형 요구사항(슬랙 메시지, 구두 요청, 메일, 회의록 등)을 분석하여 문제 정의/배경/목표/범위/리스크/결정 필요 섹션이 포함된 구조화된 Markdown 문서로 변환."`

### 2. Why를 설명하라

"ALWAYS/NEVER" 같은 강압적 지시 대신, 왜 그렇게 해야 하는지 이유를 전달한다. LLM은 이유를 이해하면 엣지 케이스에서도 올바르게 판단한다.

나쁜 예: `Options API 사용 금지`
좋은 예: `Options API를 사용하지 않는다. 타입 추론과 트리 셰이킹에 불리하기 때문이다`

### 3. Lean하게 유지하라

SKILL.md 본문은 500줄 이내를 목표로 한다. 컨텍스트 윈도우는 공유 자원이다.

- 핵심 규칙과 절차만 SKILL.md에 둔다
- 상세 레퍼런스, 예시 모음, 패턴 카탈로그는 `references/`로 분리한다
- SKILL.md에는 "언제 이 파일을 읽으라"는 포인터를 남긴다
- 300줄 이상의 reference 파일에는 상단에 목차(ToC)를 포함한다

### 4. 일반화하라

특정 예시에만 맞는 좁은 규칙보다, 원리를 설명하여 다양한 입력에 대응할 수 있게 한다.

나쁜 예: `파트너 링크 기능은 반드시 Sub-task로 분리한다`
좋은 예: `의존성 항목 중 미결 상태인 항목은 별도 Sub-task 또는 Blocker 이슈로 분리한다`

### 5. 명령형으로 작성하라

"~할 수 있다", "~하는 것이 좋다" 대신 "~한다", "~하라" 형태의 지시형 어조를 사용한다.

## template.md 작성 규칙

- template.md는 출력의 스키마 역할을 한다. 실제 출력이 이 구조를 그대로 따라야 한다
- 플레이스홀더는 `[설명]` 형태로 표기한다
- JSON 템플릿은 유효한 JSON 구조를 유지한다 (파싱 가능해야 함)
- 필수 필드와 선택 필드를 구분한다

## verify.sh 작성 규칙

- 최소한의 구조 검증만 수행한다 (필수 필드 존재 여부)
- `CONTENT_FILE` 환경 변수로 출력 파일 경로를 받는다
- 실패 시 누락된 항목을 stdout에 출력하고 비정상 종료한다
- 내용의 품질은 검증하지 않는다 (그것은 learn 스테이지의 역할)

## 품질 개선 루프

```
1. 파이프라인 실행
2. learn 스테이지가 각 출력의 품질을 분석하여 skill_patch 제안
3. 해당 챕터 담당자가 제안을 리뷰
4. PR로 SKILL.md에 반영 (Learn Agent 추가 지침 섹션)
5. 다음 실행에서 개선된 결과 확인
```

### Learn 피드백 반영 위치

learn이 생성하는 `skill_patch`는 SKILL.md 하단의 `## Learn Agent 추가 지침` 섹션에 누적된다. 이 섹션이 과도하게 길어지면 핵심 내용을 SKILL.md 본문에 통합하고 섹션을 정리한다.

## 테스트 방법

### With-skill vs Without-skill 비교

1. 동일한 입력으로 SKILL.md를 포함한 실행과 제외한 실행을 비교한다
2. 출력의 구조, 구체성, 정책 반영도를 비교하여 스킬의 부가가치를 확인한다
3. 차이가 미미하면 스킬이 충분히 구체적이지 않다는 신호다

### 골든 파일 테스트

1. 대표 입력 2~3개를 선정하여 기대 출력의 핵심 요소를 정의한다
2. 파이프라인 실행 후 기대 요소가 포함되었는지 확인한다
3. verify.sh에서 기계적으로 검증 가능한 항목은 스크립트에 추가한다
